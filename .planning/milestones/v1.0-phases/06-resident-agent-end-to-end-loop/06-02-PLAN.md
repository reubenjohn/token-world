Deliver hierarchical tick-summary compression (SIM-12) so long-running simulations don't
unbounded-grow their tick_summaries/ directory. Implements D-17 (online trigger), D-18 (batch
and epoch schema v2), and D-19 (stateless compressor hooked by SimulationEngine after each
tick write).

- Integration hook in `SimulationEngine.run_tick()` (all three paths)

- **File overlap with 06-00 (Wave 0)**: 06-00 modified engine/engine.py, engine/models.py,
  engine/__init__.py. 06-02 modifies the same files. Wave-0 lands BEFORE Wave-1 — safe.
  The only cross-reference: 06-00 added `TickResult.projected_state`. 06-02 does not touch

- **Haiku cost on compression**: one Haiku call per 100 ticks is ~$0.001; cheap.
- **Mutations.count format**: Phase 5 TickSummary stores `mutations` as `{"count": int, "list":
  [...]}`. The compressor reads `tick.get("mutations", {}).get("count", 0)`. If the format
  changes, the compressor must be updated.
- **agent_id field stub**: D-18 schema requires `agent_id: str` but TickSummary schema v1
  doesn't store actor. We stub to "unknown" in v1 and note as a future cleanup (add actor to
  TickSummary in Phase 7 alongside attention work). This is acceptable per 06-CONTEXT.md
  D-18 — the field is present; the value is sentinel.
- **Engine hook + exception swallowing**: by design, compression failures don't block ticks.
  Logger warning is the signal. If we see `tick_summaries/ticks/` growing unbounded in
  production, check logs for compressor warnings.
- **Integration with playtest runner (06-04)**: the playtest runner calls run_tick in a loop;
  compression naturally triggers when batch_size is reached. For short runs (20 turns), the
  default batch_size=100 means compression doesn't fire — acceptable.
- **No overlap with 06-01 or 06-03 files**: 06-01 only touches resident/ + cli.py + tests/
  test_resident/. 06-03 only touches tests/test_regression/. Safe Wave-1 parallel.
