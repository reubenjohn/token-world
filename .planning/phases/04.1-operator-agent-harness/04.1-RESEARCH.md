# Phase 4.1: Operator Agent Harness — Research

**Researched:** 2026-04-12
**Domain:** Claude Agent SDK (Python) + MCP tool invocation + subagent delegation + yield-driven control flow
**Confidence:** HIGH for Agent SDK API surface (verified against official docs); MEDIUM for Claude-Code-interactive ergonomics (inferred from SDK + CLAUDE.md-driven patterns); HIGH for YieldSignal shape (derived from existing seed mechanics and MechanicContext)

## Summary

Phase 4.1 is architecturally small — a few hundred LOC of operator-side orchestration — but load-bearing: it makes the "inversion of control" story in Phase 4 concrete by supplying the program that catches yields and closes the loop. Every major question has a direct answer in the `claude-agent-sdk` 0.1.58 Python API, and the dual entry-point requirement (D-05) collapses cleanly because the SDK's programmatic driver and Claude Code's interactive driver both consume the same `.mcp.json` + CLAUDE.md conventions.

The highest-leverage research findings are:

1. **The Agent SDK's `agents=` parameter + `AgentDefinition` is the literal implementation of D-01's "spawn Opus authoring subagent."** Subagents in the SDK are first-class: they get a fresh context window, a tailored system prompt, a restricted tool list, and a pinned model — all in one config block. No custom subprocess orchestration, no raw-API fallback, no manual retry loop. The retry loop D-18 worries about happens *inside the subagent's own tool-use cycle*: the subagent calls `Bash("validate-mechanic …")`, sees the failure as a ToolResult, and iterates. `max_turns` on the outer `ClaudeAgentOptions` caps the whole thing. This is dramatically simpler than a custom harness orchestrating the retry.
2. **The dual entry-point "shared MCP surface" is not a trick — the SDK and Claude Code are literally the same MCP client.** An Agent SDK Python program configured with `mcp_servers={"token-world": {"command": "uvx", "args": [...]}}` runs the exact same stdio MCP server that Claude Code launches from `.mcp.json`. Same tool schemas, same transport. The harness code path for calling `resume_tick` is identical in both modes; the only difference is who opens the outer Claude session.
3. **The operator does NOT need to implement its own MCP client.** Every pattern we'd want — "Agent SDK Python driver calls `resume_tick`" — is just "put the token-world MCP server in `mcp_servers`, allow `mcp__token-world__resume_tick`, and prompt the agent to call it." The alternative (bypass MCP, import simulation functions directly) is available but should be rejected: it creates a second, divergent call path from the one Claude Code uses, defeating D-05/D-06.
4. **The validation-retry loop (D-14, discretion D-18) is a `@tool`-wrapped `validate-mechanic` CLI invocation.** The subagent's system prompt instructs it to call `validate` after every file write; if the returned ToolResult contains failures, the subagent reads them and fixes the code. Pure LLM-driven SDLC — exactly what D-01/D-17 intends. The outer harness doesn't need a retry budget separate from `max_turns`.
5. **`YieldSignal` should be a `@dataclass(frozen=True, slots=True)` with exactly the fields Phase 5 can already produce and the operator needs to act.** Looking at `MechanicContext` (actor, target, params shape) and how seed mechanics classify actions, a minimal-but-complete shape is seven fields. Locking it now forces Phase 5 to conform; deferring creates churn when Phase 5 lands. CONTEXT.md D-07/D-10 already obligate locking.
6. **Interactive-mode "just works" is achievable via CLAUDE.md instructions alone.** Claude Code reads the universe's CLAUDE.md on session start. A "When a tick yields" section with the canonical flow (`run-tick` → read `inspect-yield` output → author mechanic → `resume-tick`) plus a single invariant ("the `mechanic-author` subagent writes the mechanic; don't write it yourself unless instructed") lets Claude Code drive the same loop autonomously. This is the D-19 default.
7. **Real-LLM tests are affordable if scoped correctly.** One end-to-end integration test that drives the full stub→Opus→validate→resume loop costs roughly one Opus mechanic-authoring session (~$0.10–$0.50 per run). At 1–3 such tests in `pytest -m integration`, marked slow and excluded from the default suite, cost-per-CI is tolerable without VCR. Default: `@pytest.mark.integration` (or `slow`) excluded from `pytest -q`; runs in a dedicated CI job or `make integration-tests`.

**Primary recommendation:** Use the SDK's `agents=` subagent parameter for the authoring step — don't build a custom subagent orchestrator. Use the SDK's `mcp_servers=` for both entry points — don't build a custom MCP client. Lock `YieldSignal` now with seven fields. Write a concrete CLAUDE.md "When a tick yields" section in the universe template so interactive mode Just Works.

### Pattern 1: `YieldSignal` — Locked Contract (D-07, D-10)

**What:** A frozen dataclass Phase 5's engine will emit verbatim and the operator will consume. The 4.1 stub fabricates instances; they match 1:1 when Phase 5 lands.

**When to use:** Emitted by the engine whenever action classification succeeds but no mechanic matches the classified action. Consumed by `OperatorHarness.handle_yield(signal)`.

**Why these fields:** Minimum information the Opus authoring subagent needs to author a correct mechanic, cross-checked against `MechanicContext` (what's available to `check`/`apply` at runtime) and seed-mechanic patterns (`actor`, `target`, how params flow in).

```python
# src/token_world/operator/yield_signal.py
from __future__ import annotations

from dataclasses import dataclass, field, asdict
from typing import Any
import json


@dataclass(frozen=True, slots=True)
class YieldSignal:
    """Contract between simulation engine and operator.

    Emitted when the engine classifies a resident-agent action but finds no
    matching mechanic in the registry. The operator authors the missing
    mechanic, writes it to mechanics/<id>.py, and calls resume_tick to
    continue the halted tick.

    Locked in Phase 4.1; Phase 5 engine produces instances of exactly this shape.
    """

    # --- Identity & provenance ---
    tick_id: str
    """The tick the engine halted on. resume_tick(tick_id=...) continues it."""

    universe_path: str
    """Absolute path to the universe folder. Disambiguates multi-universe setups."""

    schema_version: int = 1
    """Bumped on breaking changes to this dataclass."""

    # --- Why we yielded ---
    reason: str = "no_mechanic_for_action"
    """Currently the only reason. Future reasons might include
    'no_viable_action_route' (v2), but GAP-ENG16 routing of nonsense
    returns without yielding, so this stays single-valued in v1."""

    # --- What the resident agent tried to do ---
    action_text: str = ""
    """Raw free-form text the resident agent produced (e.g., 'pick up the rock').
    Unclassified. Preserved for the authoring subagent's context."""

    classified_action: dict[str, Any] = field(default_factory=dict)
    """Structured output from Phase 5 Haiku classifier. Expected keys:
        - verb: str              (e.g., 'pickup', 'move', 'speak')
        - actor: str             (node id of the agent)
        - target: str | None     (node id if applicable)
        - params: dict[str, Any] (free-form, verb-specific payload)
    The operator reads these to understand what mechanic to write."""

    # --- Context for the authoring subagent ---
    actor_state: dict[str, Any] = field(default_factory=dict)
    """Snapshot of the actor node's properties at halt. Enables the subagent
    to reason about preconditions without a second graph query."""

    candidate_mechanic_ids: list[str] = field(default_factory=list)
    """Engine's best guess at plausible existing mechanic IDs that were
    considered but ruled out (from Phase 5 matcher). Helps the subagent
    decide: extend an existing mechanic vs author a new one."""

    def to_json(self) -> str:
        """Serialise for diagnostics. Always round-trip-safe."""
        return json.dumps(asdict(self), indent=2, sort_keys=True)

    @classmethod
    def from_json(cls, payload: str) -> "YieldSignal":
        return cls(**json.loads(payload))
```

**Why seven fields (not three, not twelve):**
- `tick_id` + `universe_path` — identity; `resume_tick` needs them.
- `reason` — enum-ish; lets the harness dispatch future reason types without breaking the dataclass.
- `schema_version` — Phase 5 evolution insurance.
- `action_text` + `classified_action` — the "why we halted" surface. `action_text` preserves intent; `classified_action` is the actionable shape.
- `actor_state` — eliminates "the subagent asks which properties the actor has" by pre-answering it. Cheap; <1KB per tick.
- `candidate_mechanic_ids` — prevents the subagent from rediscovering that a `pickup` mechanic already exists and needs a parameter extension rather than a new file.

**Fields explicitly excluded (with rationale):**
- Full graph snapshot — too big; subagent can query via MCP `list_mechanics` or `Bash("cat mechanics/...")`.
- Resident agent's memory — Phase 6's concern; not needed for mechanic authoring.
- Raw LLM prompts/responses — live in Phase 4 diagnostics, not the signal itself.
- Suggested mechanic `id` — forces a naming choice from the classifier that the author should make.

### Pattern 2: `OperatorHarness` — Agent SDK Driver (D-01)

**What:** A class that wraps `claude_agent_sdk.query()` with the right options to catch a yield, spawn the mechanic-author subagent, and call `resume_tick`. One method per entry point.

**Example (skeleton; full implementation ~100 LOC):**

from __future__ import annotations

import asyncio
from dataclasses import dataclass
from pathlib import Path
from typing import Any

    AgentDefinition,
    ClaudeAgentOptions,
    ClaudeSDKClient,
    ResultMessage,
    query,
)

from token_world.operator.yield_signal import YieldSignal
from token_world.operator.subagent import build_mechanic_author_agent
from token_world.operator.validation_tool import build_validation_server
from token_world.operator.mcp_client import load_universe_mcp_config


@dataclass(frozen=True, slots=True)
class OperatorResult:
    success: bool
    tick_id: str
    mechanic_id: str | None  # The id the subagent authored, if any
    attempts: int
    final_message: str
    cost_usd: float | None = None

    """Drives the yield→author→resume loop for one tick.

    Usage (programmatic):
        harness = OperatorHarness(universe=Path("universes/my-world"))
        result = await harness.handle_yield(yield_signal)

    Usage (from CLI):
        token-world run-tick ./my-world
        (Internally instantiates this class with universe path from cwd/arg.)

        self.model = model

        mcp_servers = load_universe_mcp_config(self.universe)
        validation_server = build_validation_server(self.universe)
        mcp_servers["validation"] = validation_server

**Key design choices embedded in this skeleton:**

1. **`mcp_servers` is merged from universe `.mcp.json` + a fresh SDK MCP server for the validation tool.** This is the idiomatic way to expose both "real" simulation MCP tools and "in-process" Python wrappers in one session. [CITED: code.claude.com/docs/en/agent-sdk/mcp]
2. **`allowed_tools` includes `"Agent"`.** Without it, the subagent invocation mechanism is disabled — the parent can't spawn `mechanic-author`. [CITED: code.claude.com/docs/en/agent-sdk/subagents §Programmatic definition "The `Agent` tool must be included in `allowedTools`"]
3. **`permission_mode="bypassPermissions"` for programmatic.** The alternative (`"dontAsk"` + careful allow rules) is safer but chatty during development; the inner tools are fully whitelisted anyway so the risk surface is bounded. *Caveat:* subagents inherit `bypassPermissions` — acceptable because the mechanic-author subagent's `tools=[...]` list is explicitly scoped. [CITED: code.claude.com/docs/en/agent-sdk/permissions]
4. **`cwd` set to the universe path.** Makes the subagent's `Read`/`Write` naturally relative to the mechanics folder.
5. **`max_turns=20`** sets the outer loop budget. Empirically, authoring a simple mechanic + 1–2 validation retries fits in ~10 turns; 20 gives headroom without runaway.

### Pattern 3: Mechanic-Author Subagent Definition (D-17)

**What:** An `AgentDefinition` passed via `agents={}` that describes the Opus authoring subagent. Gets its own system prompt, tool whitelist, and model.

the validation tool.

# The yield signal

```json
{yield_json}
```

# Universe path
{universe}

1. **Understand the action.** Read `classified_action.verb`, `actor`, `target`,
   and `params`. These tell you what the agent tried to do.

2. **Check existing mechanics first.** Call `mcp__token-world__list_mechanics`.
   If `candidate_mechanic_ids` in the yield are populated, Read those files.

   or the framework docs) is the canonical reference. Skim it if you haven't
   recently.

   `<id>` is a short, descriptive snake_case name. One `Mechanic` subclass.
   Required class attributes: `id`, `description`. Default `voluntary=True`,
   `tags=[]`. Implement `check(ctx) -> CheckResult` and
   `apply(ctx) -> list[Mutation]`. See the seed mechanics for reference.

5. **Validate.** Call `mcp__validation__validate_mechanic` with the module
   path. If it returns findings, fix them and re-validate. Continue until the
   report returns `passed=True`.

6. **Write a test.** `tests/test_mechanics/test_<id>.py` following the mirror
   layout. Use the `GraphBuilder` fixture.

   - The mechanic id you authored (or extended)
   - The file path(s) you wrote
   - A one-line description of the mechanic's semantics

- Use only `MechanicContext` DSL; never import `networkx` or
  `token_world.graph.knowledge_graph` directly. The validator will reject
  such imports.
- `eval`, `exec`, `__import__`, bare `open` are forbidden in mechanic modules.
- Keep mechanics small and composable. If the action is a primitive, write a

```

**Design notes:**
- Subagents get a fresh context window per SDK docs; the yield signal is embedded in the system prompt so the subagent has everything up front without tool-call round-trips.
- Excluding `"Agent"` from the subagent's `tools` enforces "subagents cannot spawn subagents" (per SDK note).
- Pinning `model="opus"` ensures D-02 even if the outer options default to something else.

### Pattern 4: Validation Tool Wrapper (@tool)

**What:** A `@tool`-decorated async function that the subagent calls instead of shelling out to `Bash("uv run validate-mechanic …")`. Returns structured `ValidationReport` fields the subagent can reason about directly.

**Example:**
```python
# src/token_world/operator/validation_tool.py
from __future__ import annotations

from pathlib import Path
from typing import Any
import subprocess
import json

from claude_agent_sdk import create_sdk_mcp_server, tool


def build_validation_server(universe: Path):
    @tool(
        "validate_mechanic",
        "Validate a mechanic module. Returns structured report: passed, findings (list of {stage, rule, severity, message, line}). Call after every edit to a mechanic file.",
        {"module_path": str},
    )
    async def validate_mechanic(args: dict[str, Any]) -> dict[str, Any]:
        module_path = args["module_path"]
        # Reuse Phase 4's validate-mechanic CLI with --format json
        result = subprocess.run(
            ["uv", "run", "validate-mechanic", str(universe), module_path, "--format", "json"],
            capture_output=True,
            text=True,
            cwd=universe.parent,  # run from project root so uv resolves deps
        )
        try:
            report = json.loads(result.stdout)
        except json.JSONDecodeError:
            return {
                "content": [{
                    "type": "text",
                    "text": f"Validator crashed. stderr:\n{result.stderr}",
                }],
                "is_error": True,
            }

        # Prefer a clean text rendering for the LLM to read, with the full
        # structured report attached as a second content block (or serialised in text).
        return {
            "content": [{
                "type": "text",
                "text": json.dumps(report, indent=2),
            }],
            "is_error": not report.get("passed", False),
        }

    return create_sdk_mcp_server(
        name="validation",
        version="1.0.0",
        tools=[validate_mechanic],
    )
```

**Key behaviours:**
- Uses `is_error` on the ToolResult to signal validation failure without raising (keeps the agent loop alive per [CITED: code.claude.com/docs/en/agent-sdk/custom-tools §Handle errors]).
- Shells out to the real `validate-mechanic` CLI → single source of truth for validation logic.
- `--format json` requirement on the Phase 4 CLI is already listed as Claude's discretion in 04-RESEARCH.md; 4.1 depends on it being there. **Planner note:** coordinate with Phase 4 team (likely already done when 4.1 plans — but confirm plan 04-02 ships JSON output).

### Pattern 5: Dual Entry Points — How Interactive and Programmatic Share the Surface (D-05)

**The insight:** Both entry points land in the same universe directory, use the same `.mcp.json`, and invoke the same 3 MCP tools. The difference is who opens the outer Claude session:

| Aspect | Programmatic (`run-tick` CLI) | Interactive (Claude Code in universe) |
|--------|-------------------------------|---------------------------------------|
| Who opens outer session | `OperatorHarness` via `claude_agent_sdk.query()` | User types `claude` in universe folder; Claude Code starts |
| Outer system prompt | `_outer_system_prompt(signal)` | CLAUDE.md (read by Claude Code automatically) |
| Subagent invocation | `agents={"mechanic-author": ...}` passed explicitly | Instructed by CLAUDE.md to "use the `mechanic-author` subagent" — either built-in general-purpose with custom prompt, OR a filesystem agent at `.claude/agents/mechanic-author.md` |
| `mcp_servers` | Loaded from `.mcp.json` + validation server | Loaded automatically by Claude Code from `.mcp.json` |
| Yield trigger | `await harness.handle_yield(stub.emit_yield())` | Human (or future resident agent) runs `token-world run-tick` which emits yield, then Claude Code detects halt via MCP `resume_tick` response and drives from there |
| Validation tool | In-process SDK MCP server (one-shot) | User has `validate-mechanic` CLI; can be added to universe `.mcp.json` as a stdio server if we want the interactive subagent to call it via MCP too |

**Recommendation for interactive parity:** Add a filesystem-based agent at `.claude/agents/mechanic-author.md` in the scaffolded universe, with the **same prompt content** as the programmatic `AgentDefinition`. This way interactive Claude Code can "use the mechanic-author subagent" and get the identical experience. [CITED: code.claude.com/docs/en/agent-sdk/subagents §Filesystem-based definition — "Programmatically defined agents take precedence over filesystem-based agents with the same name"]

This is a small scaffolding addition: `universe/scaffold.py` writes the agent markdown file alongside `CLAUDE.md`.

def _resolve_universe(slug_or_none: str | None) -> Path:
    """Locate universe: explicit slug > env TOKEN_WORLD_UNIVERSE > CWD if it looks like a universe > error."""
    if slug_or_none:
        return UniverseManager().load(slug_or_none)
    env = os.environ.get("TOKEN_WORLD_UNIVERSE")
    if env:
        return Path(env)
    cwd = Path.cwd()
    if (cwd / ".mcp.json").exists() and (cwd / "mechanics").is_dir():
        return cwd
    raise click.ClickException(
        "No universe specified. Pass a slug, set TOKEN_WORLD_UNIVERSE, or run inside a universe folder."
    )

1. Explicit slug arg.
2. `TOKEN_WORLD_UNIVERSE` env var.
3. CWD if it has `.mcp.json` + `mechanics/` (universe marker).
4. Error with clear message.

This means a human inside a universe can type `token-world run-tick` with zero args — matching the "just works" goal.

### Anti-Patterns to Avoid

- **Importing simulation internals directly from the operator** (e.g., `from token_world.simulation import resume_tick`). Breaks D-05 dual-entry parity. Always go through MCP.
- **Custom retry loop in the outer harness** (e.g., `for attempt in range(3): ...`). The subagent's own tool-use loop handles retries; `max_turns` caps the outer session. Adding an outer retry duplicates and diverges.
- **Raw Anthropic SDK in `operator/`.** CLAUDE.md is explicit: operator is Agent SDK. Raw SDK is Phase 5's domain.
- **A new MCP tool for `inspect-yield`.** D-06 forbids it. Use CLI + filesystem reads.
- **Reading diagnostics by parsing file names with regex.** Always go through the Phase 4 `DiagnosticsSink` reader API (per D-16).
- **Hardcoding universe paths in the harness.** Always resolve via `_resolve_universe()` helper; the harness must be universe-agnostic.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Outer Claude session that spawns Opus with tools | Custom anthropic.Messages.create + tool-use loop | `claude_agent_sdk.query()` with `ClaudeAgentOptions` | SDK ships tool-use loop, hooks, budget tracking, session persistence; reimplementing is ~500 LOC of duplicated logic |
| Subagent with fresh context + pinned model + whitelist | Manual second Claude session with parent passing messages | `AgentDefinition` in `agents=` param | First-class SDK feature; context isolation, parallel execution, model override all free |
| MCP client to call token-world's `resume_tick` from Python | Custom JSON-RPC-over-stdio client | `mcp_servers=` in `ClaudeAgentOptions` | SDK already runs an MCP client; reuses Claude Code's; zero extra code |
| Wrap a CLI as a subagent-callable tool | Tell subagent to `Bash("uv run validate-mechanic …")` and parse stdout | `@tool` + `create_sdk_mcp_server` | Structured `is_error` + JSON content block the LLM can reason about; ~30 LOC |
| Filesystem layout for subagent defs | Ad hoc agent registry | `.claude/agents/<name>.md` | Claude Code native; interactive mode reads automatically |
| Retry budget for validation failures | Custom loop around subagent invocation | `max_turns` on `ClaudeAgentOptions` + subagent's own tool-use loop | One knob; retries happen where LLM has context |
| Per-tick session persistence for resumable debugging | Custom resume-id tracking | SDK's `resume=session_id` parameter | Built-in; subagent transcripts survive session compaction |
| Parse validation CLI output | Regex on `validate-mechanic` stderr | `--format json` + `json.loads` | Phase 4 already promises structured CLI output |
| Decide whether to auto-invoke authoring in interactive | Custom mode detection | `--manual` flag + CLAUDE.md instruction "auto-invoke mechanic-author subagent when a tick yields" | Two-code-path problem collapses to one flag + one instruction |

**Key insight:** Phase 4.1's entire role is to glue Agent SDK primitives together. Every custom-build candidate has a first-class SDK equivalent. The plan should be small.

## Common Pitfalls

### Pitfall 1: Forgetting `"Agent"` in `allowed_tools`
**What goes wrong:** The outer harness configures `agents={"mechanic-author": ...}` but the subagent never gets invoked. The outer Claude writes the mechanic itself (poorly, or with wrong context).
**Why it happens:** Subagents are invoked via the `Agent` tool. If `Agent` is missing from `allowed_tools`, the parent can't call it.
**How to avoid:** Always include `"Agent"` in the outer `allowed_tools`. Add an assertion in `OperatorHarness.__init__` that refuses to proceed without it.
**Warning signs:** Integration test produces a mechanic but `message.parent_tool_use_id` is None throughout — outer agent did the work itself.
[CITED: code.claude.com/docs/en/agent-sdk/subagents §Troubleshooting "Claude not delegating to subagents"]

### Pitfall 2: Subagent inherits `bypassPermissions` unexpectedly
**What goes wrong:** Outer harness uses `permission_mode="bypassPermissions"` for autonomy. Subagent's `tools=[...]` is a hint, not enforcement — it gets full system access.
**Why it happens:** Documented inheritance behaviour: "When using `bypassPermissions`, all subagents inherit this mode and it cannot be overridden."
**How to avoid:** Either (a) prefer `permission_mode="dontAsk"` with exhaustive `allowed_tools` + deny rules (restrictive but safe), or (b) accept the inheritance and rely on the subagent's `tools` whitelist being the real restriction. Since our subagent's tool list is exhaustive and file writes are scoped to the universe directory via `cwd`, option (b) is acceptable for v1.
**Warning signs:** Subagent writing outside the universe folder; calling unexpected tools.
[CITED: code.claude.com/docs/en/agent-sdk/permissions §Permission modes Warning]

### Pitfall 3: Claude Code not reading `.claude/agents/` changes mid-session
**What goes wrong:** User updates `.claude/agents/mechanic-author.md`, but the existing Claude Code session still uses the old definition.
**Why it happens:** Filesystem agents are loaded at session startup.
**How to avoid:** Document in CLAUDE.md: "If you updated `.claude/agents/`, restart the Claude Code session." Not a 4.1 bug; just user education.
[CITED: code.claude.com/docs/en/agent-sdk/subagents §Troubleshooting]

### Pitfall 4: `YieldSignal` shape drift between stub and Phase 5
**What goes wrong:** Phase 5 implements the classifier and produces a slightly-different dict shape (e.g., renames `classified_action.verb` to `action_type`). Operator code silently breaks or accepts wrong data.
**Why it happens:** The dataclass has `dict[str, Any]` fields whose contents aren't enforced.
**How to avoid:** (a) Document the required `classified_action` keys (verb, actor, target, params) in the dataclass docstring AND in a separate schema doc consumed by Phase 5. (b) Add a validator method `YieldSignal.validate()` that checks required dict keys exist. (c) Integration test in 4.1 covers the exact shape Phase 5 must produce — Phase 5 test should use the same fixture.
**Warning signs:** Phase 5 integration tests fail with `KeyError` inside the authoring subagent's context.

### Pitfall 5: Subagent recursion (subagent spawns subagent)
**What goes wrong:** Mechanic-author subagent, seeing complex work, tries to spawn its own sub-subagent.
**Why it happens:** If `"Agent"` accidentally ends up in the subagent's `tools` list.
**How to avoid:** Explicitly exclude `"Agent"` from the `AgentDefinition.tools`. The SDK docs note: "Subagents cannot spawn their own subagents. Don't include `Agent` in a subagent's `tools` array."
[CITED: code.claude.com/docs/en/agent-sdk/subagents §AgentDefinition configuration]

### Pitfall 6: Diagnostics writes race across concurrent ticks
**What goes wrong:** Phase 6's N-turn runner (future) runs ticks in parallel; two ticks yield simultaneously; their operator sessions interleave writes into shared paths.
**Why it happens:** 4.1 is scoped to single-tick; future concurrency isn't considered.
**How to avoid:** Each operator session writes into `tick_<unique_tick_id>/operator/` — tick IDs are already unique per Phase 1. No shared state. Document this invariant; verify integration test exercises two stub yields with different tick_ids.

### Pitfall 7: `resume_tick` called before mechanic file is fsync'd
**What goes wrong:** Subagent writes mechanic file, calls validation (passes), returns. Outer harness calls `resume_tick`. Registry scan happens mid-disk-flush; file not visible yet.
**Why it happens:** POSIX rename is atomic but the SDK's `Write` tool may not fsync. Windows filesystem caching exacerbates.
**How to avoid:** The `validate_mechanic` tool wrapper runs after the write and itself calls `ast.parse` on the file; if validation passed, the file is readable. This is natural synchronisation. **No extra action needed.** Document for future concurrency work.

### Pitfall 8: Validation CLI changes format between Phase 4 and 4.1
**What goes wrong:** Phase 4 plan 04-02 ships `validate-mechanic` with text-only output; 4.1's `@tool` wrapper assumes `--format json`.
**Why it happens:** Phase 4's CLI format is Claude's discretion.
**How to avoid:** **Action item for planner:** confirm with Phase 4 team (or ensure via shared planning) that `validate-mechanic --format json` ships in 04-02. If it doesn't, 4.1 plan 04.1-01 adds it as prerequisite work. Document explicitly as a Phase 4 → 4.1 contract.
