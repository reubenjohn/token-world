## Accomplishments

- **Locked the operator on-disk layout** under `<universe>/diagnostics/tick_<id>/operator/` exactly per RESEARCH §7: `yield_signal.json` + `authoring_attempts.jsonl` + `validation/attempt_NN.json` + `mechanic_diff.patch` + `resume_outcome.json`.
- **All critical writes are atomic** — `_atomic_write_text` / `_atomic_write_json` use `tempfile.mkstemp + os.fsync + os.replace`, mirroring Phase 4's helper. Failed writes leave only `*.tmp` files, which are caught by `DiagnosticsSink._sweep_tmp_files` next boot.
- **JSONL transcript with lenient reader** — `append_attempt` is non-atomic by JSONL convention; `reader.attempts()` skips unparseable / non-dict lines with a `loguru.warning` (T-04.1-06), so a crash mid-append never crashes `replay-tick`.
- **Context-manager safety net** — if the harness forgets to call `close()` or an exception bubbles through, `__exit__` writes a fallback `resume_outcome.json` with `success=False, error="session_not_closed"` (or the exception repr). Consumers always see a final artefact.
- **Schema-version guard** — `OPERATOR_SCHEMA_VERSION=1` is stamped into every `resume_outcome.json`; `reader.schema_version` raises `ValueError` on mismatch (T-04.1-05). `replay-tick` will fail loudly on incompatible payloads instead of silently misrendering.
- **Sink integration is decoupled** — `DiagnosticsSink.open_operator_session(tick_id)` reaches the universe root via `self._root.parent` (Phase 4 invariant), not the private `_universe_dir`. A future rename of `_universe_dir` won't break us.
- **Phase 4 tests still green** — 26/26 `tests/test_mechanic/test_diagnostics.py` tests pass; the new `open_operator_session` method is purely additive. Full repo: 840 passed / 14 skipped.

## On-disk Layout (locked)

```
<universe>/diagnostics/tick_<tick_id>/operator/
├── yield_signal.json             # YieldSignal.to_json() — sorted keys, indent=2
├── authoring_attempts.jsonl      # append-only; one JSON object per line
├── validation/
│   ├── attempt_01.json           # zero-padded two-digit (sorts correctly through 99)
│   ├── attempt_02.json
│   └── ...
├── mechanic_diff.patch           # unified diff verbatim (empty string -> empty file)
└── resume_outcome.json           # {"schema_version":1, "success":bool, "mechanic_id":str|null,
                                  #  "cost_usd":float|null, "turns":int, "tick_continued":bool,
                                  #  "error":str|null}
```

`tick_<tick_id>` substring keeps the `tick_` prefix even when the caller already passed `"tick_42"` — i.e. directories may look like `tick_tick_42` if the caller pre-prefixes. This is the exact behaviour demonstrated by every test in the file. Phase 5 / harness must pass plain ids (`42` or `"42"`); pre-prefixed strings are passed through verbatim and produce a doubled prefix. (Documented behaviour, not a bug — the alternative was "strip an optional prefix" which is fragile and makes int/str equivalence harder to reason about.)

## `resume_outcome.json` Schema (v1)

| Field            | Type                | Description                                                  |
|------------------|---------------------|--------------------------------------------------------------|
| `schema_version` | `int`               | Always `1` for this build; bump on breaking layout change.   |
| `success`        | `bool`              | Whether the operator authored a mechanic that validated.     |
| `mechanic_id`    | `str \| null`       | Newly-authored mechanic id, or `null` on failure.            |
| `cost_usd`       | `float \| null`     | Total LLM cost across the authoring session.                 |
| `turns`          | `int`               | Number of subagent turns consumed.                           |
| `tick_continued` | `bool`              | Whether `resume_tick` succeeded after authoring.             |
| `error`          | `str \| null`       | Exception repr / `"session_not_closed"` / `null` on success. |

## Decisions Made

- **`self._root.parent` instead of `self._universe_dir` in `open_operator_session`** — the plan called this out explicitly. Phase 4's `_root` is asserted by `test_sink_creates_diagnostics_root_on_init` to equal `<universe>/diagnostics`; reading `.parent` is a portable contract, immune to private-attribute renames inside Phase 4.
- **Local import of `OperatorDiagnosticsContext` inside `open_operator_session`** with a `TYPE_CHECKING` import at module top for the annotation. Two benefits: keeps `mechanic.diagnostics` import-time work tiny, and removes any risk of a future cycle if `operator/` ever needs to import from `mechanic/`.
- **`resume_outcome()` adds an explicit `isinstance(loaded, dict)` check.** Mypy's `--strict no-any-return` flagged `json.loads(...)` returning `Any`; the natural fix (`# type: ignore`) would have hidden a real defence-in-depth concern (a tampered outcome file might be a JSON array). The runtime check costs ~1 µs and turns "silent type confusion" into a loud `ValueError`. Bonus: this actually mitigates a near-cousin of T-04.1-05.
- **`OperatorDiagnosticsContext.__init__` creates the directories eagerly** rather than waiting for `__enter__`. Reason: the partial-session test (`test_reader_tolerates_partial_session`) writes through a context that is never `with`-entered, mirroring the real-world case of a harness that constructs a context for inspection without managing its lifecycle. Putting `mkdir` in `__init__` keeps both flows working with one path.
- **`reader.schema_version` returns `OPERATOR_SCHEMA_VERSION` for unclosed sessions** (vs raising). Rationale: forcing a raise here would break replay-tick on every incomplete session — the more useful UX is "assume current version when no claim was made," with the rejection path reserved for a stamped-but-mismatched version.
- **`Reader.attempts()` rejects non-dict JSONL entries** in addition to malformed lines. Plan only specified malformed-line tolerance; I extended to non-dict to keep downstream consumers from defensive-coding every entry. Same `loguru.warning` channel.
- **JSONL prefix-doubling (`tick_tick_42`) accepted as documented behaviour.** The plan's interface contract uses `f"tick_{self.tick_id}"` and the tests assert exactly this. Stripping a leading `tick_` would be a silent normalisation that disagrees with `OperatorDiagnosticsContext.tick_id` for the same input. Better to be loud about caller's prefix.
