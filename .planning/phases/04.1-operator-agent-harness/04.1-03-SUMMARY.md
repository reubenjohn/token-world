### Outer system prompt (verbatim — Plan 04.1-05 will use a paraphrase in CLAUDE.md)

```
You are the Token World operator. The simulation on tick {signal.tick_id!r} has halted because no mechanic matches the resident agent's classified action. The universe is at {self.universe}. Use the `mechanic-author` subagent to author the needed mechanic; it has the validation tool and will iterate until the mechanic passes validation. Once the subagent reports success, call `mcp__token-world__resume_tick` with the tick_id. Return a final JSON object on the last line of your reply: `{"success": bool, "mechanic_id": str|null, "attempts": int}`. Do NOT author the mechanic yourself — always delegate to the `mechanic-author` subagent via the Agent tool.
```

### `mechanic_author_prompt(universe, yield_json) -> str` (`src/token_world/operator/subagent.py`)

Standalone function returning the canonical authoring prompt; embedded into the subagent's `AgentDefinition.prompt` and (in Plan 04.1-05) will populate `.claude/agents/mechanic-author.md`. Includes the full yield JSON, universe path, 6-step process, and explicit AST-rule constraints (no `eval`/`exec`/`__import__`/`networkx`).

### `build_validation_server(universe)` (`src/token_world/operator/validation_tool.py`)

Returns an SDK MCP server config dict exposing the `validate_mechanic` tool. The tool wraps `uv run token-world validate-mechanic <module_path> --format json` with `shell=False` (T-04.1-11), parses the JSON report, and returns `is_error=True` when validation fails so the subagent's tool-use loop sees the failure structurally.

Companion `build_validate_mechanic_tool(universe)` exposes the underlying `SdkMcpTool` for in-process introspection (tests). The two MUST stay split: adding the tool to the server-config dict breaks the SDK's CLI subprocess transport (it JSON-serialises mcp_servers and `SdkMcpTool` is not JSON-serialisable).

### `load_universe_mcp_config(universe)` (`src/token_world/operator/mcp_client.py`)

Reads `<universe>/.mcp.json`, strips the outer `{"mcpServers": {...}}` wrapper, returns the flat dict the Agent SDK expects. Raises `FileNotFoundError` for missing files, `json.JSONDecodeError` for malformed input, `ValueError` for non-object `mcpServers`.

## Decisions Made

- **Post-ResultMessage exception swallowing.** First integration run revealed claude-agent-sdk 0.1.58 raises `Exception("Command failed with exit code 1")` AFTER yielding the `ResultMessage(subtype="error_max_turns")`. Re-raising would lose the partial result we already have. Now: if `result_received=True` when an exception fires, we log + record in `resume_outcome.json` via `error_str` but return the partial `OperatorResult` to the caller. Tests assert this behaviour.
- **Validation reports captured in the harness, not the @tool.** The @tool is a pure CLI wrapper; per-tick context is the harness's concern. `_track_validation` does a two-pass scan of message content (ToolUseBlock with name `validate_mechanic` -> assign attempt number; matching ToolResultBlock -> persist). Keeps the tool reusable and avoids coupling validation file paths into the @tool wrapper.
- **`build_validation_server` returns SDK's dict unchanged.** Initially I added `tools=[validate_mechanic]` to the returned dict for testability; this broke the SDK's CLI subprocess transport (which `json.dumps`'s the `mcp_servers` arg, and `SdkMcpTool` is not JSON-serialisable). Fix: split the API into `build_validation_server` (production) and `build_validate_mechanic_tool` (test introspection). Tested explicitly: `test_validation_server_returns_well_formed_sdk_server_config` asserts `tools` is NOT in the returned dict.
- **Outer prompt mandates delegation.** Initially I wrote a softer prompt that allowed the outer to author or delegate; revised to "Do NOT author the mechanic yourself — always delegate to the `mechanic-author` subagent via the Agent tool." Rationale: the subagent's tight `tools=[...]` whitelist (no `Agent`, no `rollback`/`resume_tick`) only matters if the outer ALWAYS uses it. Letting the outer fall back to direct authoring would make Pitfall 5's mitigation purely advisory.
- **DiagnosticsSink instantiated lazily in handle_yield.** The plan suggested `__init__` could open the sink, but tests shouldn't need a filesystem to construct a harness. Lazy instantiation also makes the harness side-effect-free until the first `handle_yield` call, which matches `_build_options` (also pure).

## Deviations from Plan

**One material deviation; everything else implemented as written.**

1. **[Rule 1 — Bug] Post-ResultMessage SDK exception handling.** The plan's harness skeleton re-raises any exception from the `query()` loop. Empirically, claude-agent-sdk 0.1.58 raises during stream cleanup AFTER yielding `ResultMessage(error_max_turns)`. Re-raising would lose the partial result the SDK already delivered. Fix: track `result_received`; on exception, if a ResultMessage already arrived, log + capture the exception in `resume_outcome.json.error` but RETURN the partial `OperatorResult` rather than re-raise. **Tracked as [Rule 1 - Bug] in this section.** Found during: integration test first run.

2. **[Rule 2 — Critical functionality missing] Validation report capture.** The plan's success criteria requires per-attempt validation reports in `operator/validation/attempt_NN.json`, but the original harness only piped messages to `authoring_attempts.jsonl`. Added `_track_validation` + `_extract_validation_report` to scan AssistantMessage content for `ToolUseBlock`/`ToolResultBlock` pairs whose tool name is `validate_mechanic`, extract the JSON report from the result, and write it. **Tracked as [Rule 2] in this section.** Found during: integration test first run (validation/ was empty).

3. **[Rule 1 — Bug] `build_validation_server` API split.** Initial version added `{"tools": [validate_mechanic]}` to the returned config dict for testability; this broke the SDK's CLI subprocess transport because the SDK JSON-serialises `mcp_servers` and `SdkMcpTool` is not JSON-serialisable. Fix: split into `build_validation_server` (production) and `build_validate_mechanic_tool` (introspection). **Tracked as [Rule 1 - Bug] in this section.** Found during: integration test first run.

4. **CLI invocation form differs from plan sketch.** Plan said `validate-mechanic --path <module>`; the actually-shipped CLI uses positional `validate-mechanic <module>` (with optional `--format json`). Verified via `uv run token-world validate-mechanic --help`. The implemented form is what works.

**Total deviations:** 4 (3 Rule-1/Rule-2 inline fixes during integration test; 1 plan-sketch correction).
