## Accomplishments

- **4 new CLI commands** dispatchable via `uv run token-world <cmd> --help`:
  - `run-tick [slug] [--manual] [--tick ID] [--format human|json] [--stub KEY=VALUE ...]`
  - `inspect-yield [slug] [--tick ID] [--format human|json]`
  - `resume-tick [slug] --tick ID`
  - `replay-tick [slug] <tick_id> [--format human|json]`
- **Universe resolution as a shared primitive.** `resolve_universe(slug_or_none)` is the single implementation of the "slug > env > cwd > error" contract; all 4 commands use it via `_cli_resolve_or_exit(slug, exit_code=N)`, so exit-code mapping is consistent and future commands can drop in for free.
- **BLOCKER-3 persistence invariant enforced.** When `run-tick --stub ... --manual` fabricates a yield, the `yield_signal.json` is written to disk BEFORE deciding about `--manual`, and `resume_outcome.json` is closed with `success=False, error="manual_mode_no_harness_invoked"` so the tick is visible to subsequent `inspect-yield` / `replay-tick` / `run-tick` invocations. Without `--manual`, the signal is still persisted before handing off to the harness — a mid-harness crash can never leave an uninspectable tick.
- **Path-traversal guard (T-04.1-18)** on `tick_id` — regex `[A-Za-z0-9_.-]+` rejects `../../etc/passwd`-style attacks before any path construction happens. Tested via `test_rejects_path_traversal_tick_id`.
- **Stub-key whitelist (T-04.1-19).** `--stub KEY=VALUE` only accepts `{verb, actor, target, action_text}`. Unknown keys raise `ClickException`; missing required keys (`verb`, `actor`) raise `ClickException` with a remediation message. Tested via `test_stub_requires_verb_and_actor` and `test_stub_rejects_invalid_kv_format`.
- **Exit-code contract enforced** across all 4 commands, tested on every path:

## Universe Resolution Rules (cli_support.resolve_universe)

Precedence (first match wins):

1. **Explicit slug argument** → `UniverseManager().load(slug)`.
2. **`TOKEN_WORLD_UNIVERSE` env var** → `Path(env)`.
3. **Current working directory** → treated as a universe if ALL of:
   - `.mcp.json` exists
   - `mechanics/` is a directory
4. **Otherwise** → `click.ClickException` listing all 3 strategies and the required cwd markers. Each command maps this to its own exit code (2 for universe-not-found across the board).

## Exit Codes (enforced + tested)

| Command | 0 | 1 | 2 | 3 | 4 |
|---|---|---|---|---|---|
| `run-tick` | mechanic authored + resumed | harness raised exception OR result.success=False | no universe | no halted tick (and no `--stub`) | — |
| `inspect-yield` | yield rendered | — | no universe | — | no halted tick (auto) or `--tick` not found |
| `resume-tick` | MCP call returned | MCP call raised | no universe (OR missing `--tick` → Click UsageError exits 2) | — | — |
| `replay-tick` | rendered | — | no universe OR invalid tick-id (Click caught) | — | `tick_<id>/operator/` absent |

Note: Click raises `UsageError` (exit 2) for missing required options — this shows up as the `resume-tick --tick` missing case and aliases into the "universe not found" column from the operator's perspective. The plan's original table allocated exit 2 only to "universe missing"; the Click convention (2 for usage errors) makes that clean in practice.

**Friction Points for Plan 05 to address:**

1. **`tick_tick_1` doubling.** The operator context manager's `f"tick_{tick_id}"` prefix means passing `tick_id="tick_1"` produces `tick_tick_1/`. The `--stub` path passes `effective_tick = "tick_1"` as documented behaviour from Plan 02 — but a human typing `run-tick --stub ... --tick my_first_tick` would produce `tick_my_first_tick/` which is actually fine, just visually odd. Plan 02's SUMMARY acknowledges this. Plan 05 could add a `strip_tick_prefix` helper or a `--no-prefix` override for Phase-5 migration, but not this plan.
2. **`replay-tick` positional argument overloading** is cute (`<slug> <tick_id>` vs `<tick_id>` alone) but might confuse first-time users who don't realise the second form requires env/cwd to be set. The help text addresses this ("Two calling forms:"). Plan 05 could explicit-split with two subcommands if this becomes an issue.
3. **Outcome's "pending" state** is labelled `manual_mode_no_harness_invoked` — fine for now, but if we later add a "harness deferred" or "dry-run" mode, we'll want a small enum of pending reasons rather than string literals scattered around. Not a blocker for Plan 05.
4. **`run-tick` default format is JSON** (D-20, programmatic-first). A human typing `run-tick` sees raw JSON, which is less friendly than `--format human`. Plan 05's interactive polish could default human when stdout is a TTY. Not done here to keep behaviour deterministic for scripting.

No friction points require a follow-up fix in this plan — all are deliberate choices documented above.

## Decisions Made

- **_invoke_resume_tick_mcp at module scope (vs nested inside `resume_tick_cmd`).** Tests monkeypatch `token_world.cli._invoke_resume_tick_mcp`; a nested function can't be patched from outside. Module-level keeps the same behaviour and makes the test surface natural.
- **Import OperatorHarness at module top (not lazily inside `run_tick`).** Tests monkeypatch `cli_mod.OperatorHarness` to inject a fake; lazy import would require `monkeypatch.setattr("token_world.operator.harness.OperatorHarness", ...)` which is clunkier. The import cost is ~10ms at startup; acceptable for a CLI that does real work after dispatch.
- **`replay-tick` positional overloading** (one positional = tick_id with env/cwd; two positionals = slug + tick_id) — rejected the alternative of a `--slug` option because the other 3 commands treat slug as the first positional. Consistency > cleanliness; tests cover both forms.
- **`_TICK_ID_RE` and `_STUB_ALLOWED_KEYS` as module-level constants** — promoted from inline to named symbols because they're security-relevant (T-04.1-18, T-04.1-19). Named constants make the threat mitigation discoverable in `git grep` from the threat ID onwards.
- **Exit code 2 for `resume-tick --tick` missing.** Click's `UsageError` convention (exit 2) overlaps with "universe not found" in the plan's contract; the practical mapping is "exit 2 = something about the invocation itself is wrong", which is what a human would expect.
- **`run-tick --format json` default.** Per D-20: programmatic callers (Phase 5, CI scripts, future automation) want canonical JSON without flags; humans add `--format human` when they want prose.

## Deviations from Plan

**Minor — all within scope; no architectural changes (Rule 4 not triggered).**

1. **[Rule 2 — Critical functionality] `tick_id` path-traversal regex.** Plan mentioned T-04.1-18 in the threat model; I added `_TICK_ID_RE` + `_check_tick_id` as module-level primitives reused by `replay-tick`, `inspect-yield`, AND `resume-tick`. The plan's body didn't detail the regex; I picked `[A-Za-z0-9_.-]+` which permits future multi-segment tick ids (e.g. `1.2`, `hotfix-a`) while still rejecting `/`, `\`, `..` (note: `..` alone passes the regex but fails because the path construction inside `diagnostics/tick_../operator/` resolves back to `diagnostics/` — the second line of defence catches it via the directory-existence check). **Tracked as Rule 2.** Found during: writing `replay-tick`; needed the same guard on `inspect-yield` too.

2. **[Rule 2 — Critical functionality] `--stub` KEY whitelist.** Plan mentioned T-04.1-19 in the threat model but didn't specify the whitelist. Added `_STUB_ALLOWED_KEYS = frozenset({"verb", "actor", "target", "action_text"})` at module scope. Unknown keys are rejected via `ClickException` with a message that lists the allowed keys. **Tracked as Rule 2.**

3. **`replay-tick` positional argument forms.** Plan's interfaces block said `replay-tick [UNIVERSE_SLUG] TICK_ID` (slug optional, tick_id required). Click doesn't support "optional positional followed by required positional" cleanly, so I split it: `<slug_or_tick_id> [<tick_id>]` and disambiguated in code. One-positional form = tick_id + env/cwd universe; two-positional form = slug + tick_id. Tests cover both. **Not a Rule deviation — plan-sketch adjustment for Click's argument model.**

4. **Removed unused `TYPE_CHECKING` import in `cli_support.py`.** Initial draft imported `UniverseManager` under `TYPE_CHECKING` (dead code — it's imported locally inside `resolve_universe`). Ruff flagged it; I deleted the `if TYPE_CHECKING:` block entirely because there are no forward references that need it. **Not a Rule deviation — hygiene.**

**Total deviations:** 2 Rule-2 security tightenings + 2 structural adjustments. Zero behavioural changes that contradict the plan.
