# Phase 4.1: Operator Agent Harness — Context

**Gathered:** 2026-04-12
**Status:** Ready for planning
**Source:** In-conversation discussion (see session on 2026-04-12 where Phase 4.1 was inserted)

> **Inserted phase note:** Phase 4.1 was inserted after roadmap validation surfaced that Phase 5's success criteria — "when no mechanic matches, the engine halts the tick with a structured yield signal; after the operator authors + validates the needed mechanic, a subsequent `resume_tick` picks up the new mechanic and completes the tick" — cannot be verified end-to-end without the operator-side orchestration. Phase 5's `Depends on` was updated from Phase 4 to Phase 4.1 to reflect this.

<domain>
## Phase Boundary

Deliver the **operator agent harness**: the Agent SDK-powered driver that, in response to a simulation tick yielding for a missing mechanic, authors the needed mechanic (Opus via the Agent SDK), validates it through Phase 4's pipeline, and resumes the tick. The harness must work identically whether driven interactively (a human using Claude Code inside a universe folder) or programmatically (an Agent SDK Python program, e.g. a future playtest runner).

Concretely:

1. A **one-tick operator loop**: `tick_stalled(yield_signal) → spawn Opus authoring subagent → mechanic written to disk → Phase 4 validation pipeline → resume_tick`.
2. A **harness-agnostic MCP surface** — the existing 3 simulation tools (`resume_tick`, `rollback`, `list_mechanics`) plus anything 4.1 surfaces for the operator are equally callable from Claude Code and from an Agent SDK driver.
3. **Dev UX inside a universe** — CLI commands + in-session surfaces (slash commands if appropriate) for `run-tick`, `inspect-yield`, `replay-tick` so the human operator flow is first-class.
4. **Self-verification via a throwaway engine stub** — a minimal fixture that emits a fabricated yield signal for a known missing mechanic; the harness completes the loop against this stub in automated tests. The stub is deleted when Phase 5 lands and the real engine drives the harness.
5. **Diagnostics surface** — expose Phase 4's `DiagnosticsSink` records to the operator (view yields, replay prompts/responses/mutations). CLI-first; no UI.

Explicitly OUT of scope for Phase 4.1:
- The simulation engine itself (Phase 5): action classification, mechanic matching from text, observation synthesis, conservation enforcement.
- N-turn playtest runner, quality scoring, regression suite (Phase 6; builds on the Phase 4.1 harness).
- The resident agent / personality-driven agent inhabiting the world (Phase 6).
- Multi-agent coordination (v2).
- Sandboxing of authored mechanic code (v2; Phase 4 already decided no RestrictedPython for v1).

</domain>

<decisions>
## Implementation Decisions

### Architectural Role
- **D-01:** The "operator" is a program, not a role. Concretely: an Agent SDK (`claude-agent-sdk`) Python program that owns the outer loop, spawns an Opus subagent with mechanic-authoring instructions + tool access (file writes, validation CLI), catches yield signals from the simulation tools, and drives the subagent until a valid mechanic exists on disk, then calls `resume_tick`. This is the concrete realisation of the Hybrid SDK decision in PROJECT.md §Key Decisions.
- **D-02:** Operator model = Opus (inherited from PROJECT.md; no override). Engine-layer LLM calls (Haiku classifier, Sonnet observer) remain Phase 5's concern and use the raw Anthropic SDK. Phase 4.1 does not introduce new model choices.

### Scope Boundary with Phase 6
- **D-03:** Phase 4.1 delivers the **single-tick** yield→author→validate→resume loop. Phase 6 builds the **N-turn playtest runner** on top: running many actions, scoring quality, adversarial inputs, regression suite. The dividing line is "one yield handled end-to-end" vs "an agent session runs for many turns against this harness". This avoids the harness bloating into a playtest framework and keeps Phase 6's scope intact.
- **D-04:** Phase 4.1 does NOT author the resident agent. The harness is driven in tests by a stub caller; Phase 6 supplies the real resident agent.

### Entry Points — Dual First-Class
- **D-05:** Both entry points are first-class and share the same MCP tool surface:
  - **Interactive:** A human opens Claude Code inside a universe folder. The universe's CLAUDE.md + `.mcp.json` expose the 3 simulation tools. When a tick yields, Claude Code drives the operator flow directly (human-in-the-loop review is natural here).
  - **Programmatic:** An Agent SDK Python program (importable from `token_world.operator`) drives the same MCP tools via the SDK. Used by automated tests in 4.1 and by Phase 6's playtest runner later.
- **D-06:** The 3-tool MCP surface (`resume_tick`, `rollback`, `list_mechanics`) is the stable contract. Any additional operator-side operations (view yields, replay diagnostics) are CLI commands, not MCP tools — they are developer ergonomics, not simulation primitives. Rationale: the MCP surface defines what any coding harness (Codex, Cursor, etc.) sees; keep it minimal per Phase 0 UNIV-03.

### Yield Protocol
- **D-07:** The yield signal is a structured Python object (dataclass) emitted by the simulation tools when no mechanic matches a classified action. Phase 4.1 defines the canonical shape; Phase 5's engine produces instances of it. Minimum fields: tick id, classified action, reason (`"no_mechanic_for_action"`), serialised action context (actor, target, params) sufficient for the operator to understand what mechanic to author. The operator subagent receives the yield as a structured prompt; humans see a formatted CLI rendering.
- **D-08:** `resume_tick` is idempotent w.r.t. registry state — after the operator writes a new mechanic, `resume_tick` causes the registry to re-scan, pick up the new mechanic, match against the stalled action, and continue. The registry auto-scan is Phase 4 D-15; Phase 4.1 relies on it.

### Self-Verification — Engine Stub
- **D-09:** Phase 4.1 ships a minimal `EngineStub` class (test-only, in `tests/test_operator/` or similar) that emits a fabricated yield signal matching the D-07 shape. Harness integration tests use this stub to exercise the full loop: stub-emits-yield → operator spawns Opus → Opus writes mechanic → validation passes → `resume_tick` returns success. The stub is explicitly throwaway; when Phase 5 lands, the real engine replaces it and these tests adjust to drive through the real classifier.
- **D-10:** Stub design constraint: it MUST match the exact `YieldSignal` dataclass shape Phase 5 will produce, so swapping in the real engine changes no operator code. The shape is the contract.

### Dev UX Scope
- **D-11:** CLI commands added inside a universe (or as `token-world <universe-slug> <cmd>` from outside):
  - `run-tick` — execute one tick against the universe's simulation (delegates to MCP / engine; may yield)
  - `inspect-yield` — show the currently-stalled yield for a universe (reads diagnostics substrate)
  - `resume-tick` — explicit resume (normally invoked automatically after authoring; useful for debugging)
  - `replay-tick <tick-id>` — render a past tick's diagnostics (prompts, mutations, observations) for human review
- **D-12:** Slash commands inside Claude Code are out of scope for 4.1 unless trivially cheap — the CLI + MCP tools are enough surface. Can be added later by any universe's CLAUDE.md template without framework changes.

### Testing Model
- **D-13:** Harness tests are real LLM integration tests by default (using Opus via Agent SDK). Cost and flakiness are acceptable costs for the feedback-loop-closing that this phase is entirely about. A VCR-style recording fixture may be added if tests become unacceptably slow; defer the decision to the planner.
- **D-14:** Harness tests consume Phase 4's validation pipeline as-is — no mocking the validator. If Opus writes invalid code, the harness must surface the validation failure back to the subagent (as a prompt) and retry up to N times (small N; planner picks).

### Diagnostics Integration
- **D-15:** The harness writes operator-level diagnostics to the same per-tick folder Phase 4 defines — specifically an `operator/` subfolder with: yield signal snapshot, authoring subagent prompts + responses, validation reports per attempt, final mechanic diff, resume_tick outcome. Extends Phase 4 D-21/D-22 schema with an operator namespace.
- **D-16:** `replay-tick` CLI (D-11) renders the operator subfolder human-readably. Re-uses the Phase 4 diagnostics reader API; does not reimplement parsing.

### Claude's Discretion
- **D-17:** Exact structure of the Agent SDK subagent system prompt (framework-level authoring guide vs universe-specific guidance vs both).
- **D-18:** Retry policy for failed authoring attempts (number of retries, feedback format back to subagent).
- **D-19:** Whether the interactive (Claude Code) path auto-invokes the authoring subagent or requires the human to explicitly author. Reasonable default: auto-invoke, but expose a `--manual` flag on `run-tick` to suppress it.
- **D-20:** CLI output formatting and JSON vs human-readable default for `inspect-yield` / `replay-tick`.
- **D-21:** Whether the engine stub lives in `tests/` or `src/token_world/operator/testing/` — pick based on what's easiest to import from tests.

### Proposed Plan Decomposition (informative; planner has final say)
1. **04.1-01 — Yield protocol + engine stub.** Define `YieldSignal` dataclass; write `EngineStub` that emits fabricated yields; unit tests for the shape and serialisation.
2. **04.1-02 — Operator core (Agent SDK driver).** `token_world.operator.OperatorHarness`: opens an Agent SDK session with Opus, receives a yield signal, spawns mechanic-authoring subagent with tool access, loops on validation failure, calls resume. Integration test against the engine stub: end-to-end one-tick loop.
3. **04.1-03 — Dev UX CLI.** `run-tick`, `inspect-yield`, `resume-tick`, `replay-tick` — implemented against the operator harness API and the Phase 4 diagnostics reader.
4. **04.1-04 — Diagnostics operator namespace.** Extend Phase 4 `DiagnosticsSink` schema with operator subfolder; wire harness writes; `replay-tick` reads and renders.
5. **04.1-05 — Interactive entry point polish.** Universe CLAUDE.md template additions explaining the yield flow for a human driver; any small MCP adjustments discovered during 04.1-02; smoke test that Claude Code interactive mode can complete the same loop a programmatic driver does.

</decisions>

## Existing Code Insights

### Reusable Assets
- **MCP server stub** (Phase 0) — already exposes the 3-tool surface; Phase 4.1 calls these from the programmatic driver.
- **Validation pipeline** (Phase 4) — the harness invokes it as a library call + CLI; no reimplementation.

### Gaps the Harness Must Fill
- No code today catches yield signals (no yield signals exist yet — 4.1 defines the shape).
- No code spawns a mechanic-authoring subagent.
- No code uses Agent SDK at all yet (the package is a new dependency for 4.1).
- No CLI commands exist for `run-tick` / `inspect-yield` / `replay-tick`.

<specifics>
## Specific Ideas from Discussion

- The human-driver case is important dogfooding for "open Claude Code in a universe folder and it Just Works" — the operator CLI + MCP surface + CLAUDE.md guidance should be good enough that a hobby user doesn't have to run the programmatic driver.
- The operator harness is the missing link the original roadmap glossed over by saying "yields to the operator" without specifying what the operator *is*.
- Self-improving infrastructure principle (CLAUDE.md §4) applies here: the dev tools built for simulation playtesting are the same tools agents use to build the project. No separate "dev tools" vs "simulation tools".

</specifics>
