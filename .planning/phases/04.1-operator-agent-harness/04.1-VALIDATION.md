## Manual-Only Verifications

The interactive smoke test is the single manual gate. It proves CONTEXT Success Criterion #2 — *"the same yield→author→resume loop completes from an interactive Claude Code session inside a universe folder."* Both the CLI and interactive paths share the same mechanic-author prompt (T-04.1-22 mitigation) so functional parity is structurally guaranteed; the smoke test is an end-to-end sanity check, not a parity-of-behaviour check.

| Behavior | Requirement | Why Manual | Test Instructions |
|----------|-------------|------------|-------------------|
| Claude Code reads CLAUDE.md Operator Flow and drives yield → author → resume autonomously | D-05 (interactive entry point parity), UNIV-03 | Requires a live Claude Code session; automatable only partially via the programmatic path (which is covered by Plan 03 Task 3's integration test) | See steps below |

### Smoke test steps

1. Export an API key: `export ANTHROPIC_API_KEY=sk-ant-…` (Claude Code will prompt for login if missing).
2. Create a fresh universe: `uv run token-world create test-operator-interactive`.
3. `cd` into the created universe folder (printed by step 2).
4. Seed a halted tick. Until Plan 04's `run-tick --stub` flag lands, the simplest way is to construct a stub yield manually (programmatic):
   ```bash
   uv run python -c "
   from pathlib import Path
   from token_world.operator.testing import EngineStub
   from token_world.operator.diagnostics import OperatorDiagnosticsContext
   universe = Path.cwd()
   signal = EngineStub(universe).fabricate_yield(verb='dig', actor='alice')
   ctx = OperatorDiagnosticsContext(universe, signal.tick_id)
   ctx.write_yield_signal(signal)
   print('seeded yield at', ctx.operator_dir / 'yield_signal.json')
   "
   ```
   (Once Plan 04 lands: `uv run token-world run-tick --manual --stub verb=dig --stub actor=alice` does the same.)
5. Launch Claude Code inside the universe folder: `claude`.
6. Prompt Claude Code exactly: *"I just ran run-tick and got a yield. Please handle it per the Operator Flow in CLAUDE.md."*
7. Observe the flow:
   - Claude Code Reads CLAUDE.md's Operator Flow section.
   - Claude Code invokes the `mechanic-author` subagent (via the Agent tool on the outer session).
   - The subagent reads the authoring guide, writes `mechanics/<id>.py`, and iterates against `mcp__validation__validate_mechanic` until `passed=true`.
   - Claude Code calls `token-world resume-tick --tick tick_1` (once Plan 04 ships — today the programmatic harness is the equivalent resume path).
8. Pass criteria:
   - (a) `mechanics/<id>.py` exists and imports cleanly.
   - (b) `token-world replay-tick tick_1` shows `success=True` (or `OperatorDiagnosticsReader(universe, "tick_1").resume_outcome()["success"] is True` while Plan 04 is in flight).
   - (c) No human code-writing was required.
   - (d) **Observed cost recorded in the smoke-test row above; flag if greater than $5.** Cost-cap enforcement lives in `OperatorHarness` (`max_budget_usd=5.0` → SDK hard cap per Plan 03 BLOCKER-4 resolution). The smoke test's job is to observe and record, not to re-enforce.
9. Fail handling: on failure, file a gap via `/gsd-plan-phase --gaps`, capture the session transcript + diagnostics folder, and iterate on the mechanic-author prompt or CLAUDE.md guidance.
