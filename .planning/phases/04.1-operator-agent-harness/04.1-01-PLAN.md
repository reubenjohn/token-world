class YieldSignal:

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Disk → operator (`YieldSignal.from_json`) | Diagnostics files written by prior runs (or later, by Phase 5) are read back by `inspect-yield` / `replay-tick`. Untrusted in the sense that malformed or tampered JSON must not crash the operator or produce a wrong-shape object the authoring subagent trusts. |
| Test fixtures → production code | The stub is technically "production" code (ships in src/) but is test-only in intent. Must not leak into non-test code paths. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04.1-01 | Tampering / Information Disclosure | `YieldSignal.from_json` reading disk-persisted signals | mitigate | `from_json` rejects (a) non-dict root, (b) missing required fields (TypeError via `cls(**data)`), (c) unknown `schema_version` (ValueError). Tests in Task 2 assert each rejection path. This blocks silently accepting a payload whose shape no longer matches the dataclass. |
| T-04.1-02 | Spoofing | Test-only stub leaking into production flow | accept | `EngineStub` lives in `src/token_world/operator/testing.py` but is NOT exported from `operator/__init__.py`; import requires the explicit `.testing` path. Low risk because Phase 5 will remove/replace the stub entirely. |
| T-04.1-03 | Denial of Service | Unbounded nested dict in `classified_action` / `actor_state` | accept | These fields are `dict[str, Any]`; no size cap. Rationale: emitter is Phase 5's engine (trusted); stub constructs bounded values. If Phase 5 ever emits pathological payloads, revisit with a size/depth cap in `validate()`. |
| T-04.1-04 | Tampering | JSON serialisation ambiguity | mitigate | `to_json` uses `sort_keys=True, indent=2`; deterministic output makes diff-based review (diagnostics) meaningful and prevents silent-change tampering being missed. |
</threat_model>
