## Warnings

### WR-01: Final-message JSON parser fails on nested objects

**File:** `src/token_world/operator/harness.py:476`
**Issue:** `_parse_final` uses the regex `\{[^{}]*\}` to find candidate JSON blocks in the outer model's final message. The character class `[^{}]` explicitly excludes braces, so this regex *cannot* match a JSON object containing nested objects (the outer model's own contract from the system prompt is `{"success": bool, "mechanic_id": str|null, "attempts": int}` so it's flat — but the moment the model wraps its summary in prose like `"I called validate with params {duration: 5} and got {success: true, ...}"` or includes a `params` field, only the innermost `{...}` matches and the parser silently fails to find the real summary.

Confirmed empirically: against `{"success": true, "params": {"x": 1}, "attempts": 3}`, the regex matches only `{"x": 1}`. The fallback path then logs `"Could not parse final OperatorHarness JSON"` and returns `(False, None, 0)` — turning a successful run into a reported failure on the wire.

**Fix:** Either switch to a balanced-brace scanner, or use the existing `stripped = text.strip(); candidates.append(stripped)` plain-text path more aggressively first (it already handles flat JSON) and gate the regex behind a more robust extractor. Concrete fix using `json.JSONDecoder.raw_decode`:

```python
@staticmethod
def _parse_final(text: str) -> tuple[bool, str | None, int]:
    decoder = json.JSONDecoder()
    candidates: list[dict[str, Any]] = []
    # 1. Try plain parse on the full strip.
    stripped = text.strip()
    if stripped:
        try:
            obj = json.loads(stripped)
            if isinstance(obj, dict):
                candidates.append(obj)
        except (json.JSONDecodeError, ValueError):
            pass
    # 2. Scan for any JSON object embedded in prose using raw_decode,
    #    which respects brace nesting.
    i = 0
    while i < len(text):
        idx = text.find("{", i)
        if idx == -1:
            break
        try:
            obj, end = decoder.raw_decode(text, idx)
            if isinstance(obj, dict):
                candidates.append(obj)
            i = idx + end
        except json.JSONDecodeError:
            i = idx + 1
    # Prefer the LAST candidate (LLMs tend to put the summary at the end).
    for parsed in reversed(candidates):
        success = bool(parsed.get("success", False))
        mechanic_id = parsed.get("mechanic_id")
        if mechanic_id is not None and not isinstance(mechanic_id, str):
            mechanic_id = None
        attempts_val = parsed.get("attempts", 0)
        attempts = int(attempts_val) if isinstance(attempts_val, int | float) else 0
        return success, mechanic_id, attempts
    if text:
        logger.warning(...)
    return False, None, 0
```

This also adds a regression test:
```python
async def test_handle_yield_handles_nested_json_in_final(...):
    fake_msgs = [_make_result_msg(
        result='{"success": true, "mechanic_id": "x", "params": {"k": 1}, "attempts": 1}'
    )]
    ...
    assert result.success is True
```

### WR-02: `_invoke_resume_tick_mcp` swallows tool errors silently

**File:** `src/token_world/cli.py:775-779`
**Issue:** The `async for _msg in query(...)` loop discards every message including any `ResultMessage` carrying `subtype="error_..."`, `is_error=True`, or a tool-failure `ToolResultBlock`. The CLI then prints `"Resumed tick {tick_id}"` unconditionally as long as no Python exception escaped. An MCP server that returns a structured error (today's `token-world-mcp` returns `"not yet implemented"` for `tools/call` per `test_integration.py` docstring) will look identical to a successful resume from the user's perspective.

This matters for Phase 5 hand-off: when the real `resume_tick` lands, an MCP-side failure (graph corruption, invalid tick_id, etc.) will be silently masked. Operators running `token-world resume-tick --tick X && next-step` will treat failures as successes.

**Fix:** Inspect the message stream for `ResultMessage` with `is_error=True` or `subtype.startswith("error_")` and propagate as a non-zero exit:

```python
async def _invoke_resume_tick_mcp(universe: Path, tick_id: str) -> None:
    from claude_agent_sdk import ClaudeAgentOptions, ResultMessage, query
    ...
    error_subtype: str | None = None
    is_error = False
    async for msg in query(prompt=prompt, options=options):
        if isinstance(msg, ResultMessage):
            if getattr(msg, "is_error", False):
                is_error = True
            subtype = getattr(msg, "subtype", None)
            if isinstance(subtype, str) and subtype.startswith("error_"):
                error_subtype = subtype
    if is_error or error_subtype:
        raise RuntimeError(
            f"resume_tick MCP call failed (subtype={error_subtype})"
        )
```

The caller at `cli.py:971-974` already wraps in `try/except Exception` and exits 1, so raising will route correctly through the existing error path. Without this change the documented exit code contract on line 689 (`resume-tick: 0 success | 1 MCP failure`) cannot actually fire on MCP-tool-level failures.

### WR-03: `_track_validation` silently drops uncorrelated `ToolResultBlock`s

**File:** `src/token_world/operator/harness.py:402-417`
**Issue:** When a `ToolResultBlock` arrives without a matching `ToolUseBlock` in `pending` (e.g., the SDK emits the result block in a different message-iteration order than expected, or a different SDK version reorders them), the block is silently skipped on line 405-406:
```python
if not isinstance(tool_use_id, str) or tool_use_id not in pending:
    continue
```
Because `_track_validation` is the *only* writer for `validation/attempt_NN.json`, a missed correlation means a real validation report never lands on disk. The integration test asserts `val_files, "No validation reports written"` — it would still pass as long as ONE attempt correlates, but missing intermediate reports leaves the diagnostic record incomplete. Worse, no log line is emitted, so when a forensic operator opens `replay-tick` and sees zero validation reports, there's no signal that the harness saw a result it couldn't match.

**Fix:** Add a debug-level log when an uncorrelated `validate_mechanic`-shaped result is seen, so operators can spot it in trace logs without crashing the harness:

```python
elif block_cls == "ToolResultBlock":
    tool_use_id = getattr(block, "tool_use_id", None) or getattr(block, "id", None)
    if not isinstance(tool_use_id, str):
        continue
    if tool_use_id not in pending:
        # Could be an unrelated tool's result OR a correlation miss; log
        # only when the content shape suggests a validate_mechanic report.
        raw = getattr(block, "content", None)
        if isinstance(raw, list | str):
            logger.debug(
                "Uncorrelated ToolResultBlock id={} (no pending validate_mechanic)",
                tool_use_id,
            )
        continue
    ...
```

The fix is intentionally minimal: tighten observability without changing behavior. A more aggressive fix (fall back to "claim the next pending slot") risks miscorrelating distinct validations when they overlap.

### IN-03: `_check_tick_id` regex permits `.` and `..` after prefix-strip

**File:** `src/token_world/cli.py:700` and `cli_support.py:283-292`
**Issue:** `_TICK_ID_RE = re.compile(r"^[A-Za-z0-9_.\-]+$")` accepts `..`, `.`, `...` etc. Path construction is safe in practice because the CLI always prepends the literal string `tick_` (so `..` becomes the folder name `tick_..`, not a traversal). But `latest_halted_tick` strips `tick_` before returning, so `latest_halted_tick(universe)` could return literally `..` if a folder named `tick_..` exists, which is then re-passed through `_check_tick_id` (passes!) and used as a path segment. Still benign because of the re-prefix, but the contract reads ambiguously.

**Fix:** Either (a) document the always-re-prepend invariant inline at `_TICK_ID_RE`, or (b) tighten the regex to disallow leading dots:
```python
_TICK_ID_RE = re.compile(r"^[A-Za-z0-9_][A-Za-z0-9_.\-]*$")
```
Tests already cover this with `test_rejects_path_traversal_tick_id` (rejects `../../etc/passwd` because of `/`); the dot-only case is uncovered. Add:
```python
def test_dot_only_tick_id_rejected(...):
    result = _invoke_cli(["replay-tick", ".."], universe=universe)
    assert result.exit_code != 0
```

### IN-04: `EngineStub` imported at module top in `cli.py`

**File:** `src/token_world/cli.py:23`
**Issue:** `from token_world.operator.testing import EngineStub` is at module top-level. The operator `__init__.py` docstring (lines 14-17) explicitly states EngineStub is testing-only and "deliberately NOT re-exported here to keep the production API free of test helpers." But the CLI does import it for the production `--stub` flag of `run-tick`. This is contradictory — either EngineStub is a production helper (and should be exposed) or it's test-only (and `--stub` should not depend on it).

The functional impact is zero (EngineStub works fine), but the architectural intent is muddled: the comment in `__init__.py` no longer matches reality once `cli.py` imports the helper.

**Fix:** Either:
- Move the import inside `_build_stub_signal` (matches the deferred-import pattern used elsewhere in `cli.py` for `UniverseManager`, `MechanicRegistry`, etc.), and update the `__init__.py` docstring to acknowledge `--stub` uses it; OR
- Promote `EngineStub` from `operator.testing` to a first-class public API in `operator/__init__.py` since the production CLI depends on it.

Pragmatically: the deferred-import option is one line of change and preserves the testing/production boundary cleanly.

### IN-05: Phase 4 `_atomic_write_json` and Phase 4.1 `_atomic_write_text` are near-duplicates

**File:** `src/token_world/operator/diagnostics.py:74-99` and `src/token_world/mechanic/diagnostics.py:47-76`
**Issue:** The two atomic-write helpers differ only in the payload type (str vs JSON-serializable obj). The Phase 4.1 file has `_atomic_write_text` and a one-line `_atomic_write_json` wrapper; Phase 4 has only a JSON variant. The two implementations have drifted twice (suffix patterns, parent.mkdir call ordering), and a future tweak (e.g., adding `O_DIRECT` or changing the temp-dir strategy) will need to touch two files.

**Fix:** Hoist `_atomic_write_text` to a shared module — `token_world/_io.py` or `token_world/mechanic/atomic.py` — and have both diagnostics files import it. Phase 4's existing `_atomic_write_json` becomes a one-liner like the Phase 4.1 version. Three callsites total; refactor is mechanical.

This isn't urgent (both versions are correct as written) but the ROI is high: two atomic-write implementations is a smell that grows worse with each Phase that needs persistence.
