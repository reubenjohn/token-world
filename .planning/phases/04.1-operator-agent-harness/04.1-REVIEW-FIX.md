### WR-01: Final-message JSON parser fails on nested objects

**Files modified:** `src/token_world/operator/harness.py`, `tests/test_operator/test_harness.py`
**Commits:**
- `d7e8aa8` test(04.1): WR-01 add regression tests for nested-JSON _parse_final
- `8680940` fix(04.1): WR-01 parse nested-JSON final messages via raw_decode

**Applied fix:** Replaced the naive `\{[^{}]*\}` regex in `OperatorHarness._parse_final` with a `json.JSONDecoder.raw_decode` scanner that honours brace nesting. The new parser:

1. First tries `json.loads` on the entire stripped text (the happy path when the outer model complies with the prompt's JSON-only instruction).
2. Falls back to walking every `{` in the text and calling `raw_decode` — which returns both the decoded object AND an end index, so the loop can continue picking up subsequent objects in prose.
3. Returns the LAST successfully decoded dict (LLMs put the summary last).
4. Removes the now-unused `import re`.

Added 6 tests covering: nested JSON (flat+embedded+in-prose), last-candidate preference with multiple objects, prose-only failure, and an end-to-end `handle_yield` regression.

**Verification:** Tier 1 (re-read), Tier 2 (`python -c "import ast..."` syntax-parse), Tier 3 (full pytest run — 958 passed, 14 skipped, 1 deselected).

### WR-02: `_invoke_resume_tick_mcp` swallows tool errors silently

**Files modified:** `src/token_world/cli.py`, `tests/test_cli/test_operator_commands.py`
**Commits:**
- `6080622` test(04.1): WR-02 add regression tests for MCP error propagation
- `9ce596b` fix(04.1): WR-02 propagate MCP errors from _invoke_resume_tick_mcp
- `0cf32a3` style(04.1): ruff format on new WR-02 test helper

**Applied fix:** The old `async for _msg in query(...): pass` loop discarded every streamed message, so `ResultMessage` instances carrying `is_error=True` or `subtype="error_*"` were invisible. The fix:

1. Iterates the stream and inspects each `ResultMessage`.
2. Tracks two error signals: `is_error` (bool, any True persists) and `error_subtype` (string starting with `error_`).
3. Raises `RuntimeError` after the stream completes if either signal fired.
4. Rebound `query` from a local import to `claude_agent_sdk.query` attribute access so tests can monkeypatch the module attribute without going through a real SDK subprocess.

The existing `try/except Exception` at `cli.py:972-974` converts exceptions to exit code 1 + stderr, so the documented `resume-tick: 0 success | 1 MCP failure` contract now actually fires on MCP-tool-level failures.

Added 4 tests in new `TestInvokeResumeTickMcp` class: success path (no raise), `is_error=True` raises, `subtype="error_max_turns"` raises, end-to-end CLI exit code 1.

**Verification:** Tier 1 (re-read), Tier 2 (syntax-parse), Tier 3 (full pytest + ruff + mypy on src/token_world/).

### WR-03: `_track_validation` silently drops uncorrelated `ToolResultBlock`s

**Files modified:** `src/token_world/operator/harness.py`, `tests/test_operator/test_harness.py`
**Commits:**
- `a7b192a` test(04.1): WR-03 add regression test for uncorrelated ToolResultBlocks
- `5dcda86` fix(04.1): WR-03 log uncorrelated ToolResultBlocks at DEBUG level

**Applied fix:** Minimal observability fix per the review's explicit guidance — tighten signal without changing behaviour. When a `ToolResultBlock` arrives with a `tool_use_id` not in `pending`, emit `logger.debug("Uncorrelated ToolResultBlock tool_use_id={} (no pending validate_mechanic); skipping", tool_use_id)`. Forensic operators opening `replay-tick` can now grep debug traces for correlation misses. The aggressive alternative ("claim the next pending slot") was explicitly rejected in the review — it risks miscorrelating distinct overlapping validations.

Added 2 tests:
- Orphan `ToolResultBlock` produces a debug log mentioning its `tool_use_id`, counter unchanged, no write. Uses the loguru->stdlib propagation pattern from `tests/test_graph/test_spatial_index.py:90-116` so pytest's `caplog` sees loguru records.
- Regression guard: matched `ToolUseBlock`+`ToolResultBlock` pair still writes `validation/attempt_NN.json` (the happy path is unchanged).

**Verification:** Tier 1 (re-read), Tier 2 (syntax-parse), Tier 3 (full pytest + ruff + mypy).
