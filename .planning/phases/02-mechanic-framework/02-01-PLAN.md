  - For `PropertyChangeMatcher`: mutation.type == "set_property" and mutation.property == matcher.property_name. If matcher.node_type is set, check graph.query(mutation.target).get("type") == matcher.node_type.

**engine.py** (per D-09, D-10):
- `ChainExecutionEngine` class with `__init__(self, involuntary_mechanics: list[Mechanic], max_depth: int = 10)`
- Method `execute(self, mechanic: Mechanic, ctx: MechanicContext) -> ExecutionTrace`:
  1. Call mechanic.check(ctx). If not passed, return trace with empty mutations and no children.
  2. Call mechanic.apply(ctx) to get mutations.
  3. Build root TraceNode.
  4. Call `_evaluate_chain(mutations, ctx.actor, depth=1, seen={(mechanic.id, ctx.target)})` to get children.
  5. Attach children to root. Count total mechanics. Return ExecutionTrace.
- Private method `_evaluate_chain(self, mutations, actor, depth, seen) -> tuple[list[TraceNode], bool]`:
  1. If depth > max_depth: return ([], True) for truncated.
  2. For each involuntary mechanic, check if any of its watches() matchers match any mutation (using `matches()` from matchers.py).
  3. For matching mechanic: determine target from the triggering mutation (mutation.target; for edge mutations like "src->dst", use the source node -- split on "->").
  4. Skip if (mechanic.id, target) in seen (cycle detection).
  5. Create MechanicContext with same graph, actor=actor, target=target.
  6. Call check(). If passed, call apply() to get new_mutations.
  7. Add (mechanic.id, target) to seen.
  8. Recursively _evaluate_chain with new_mutations, depth+1, seen.
  9. Build TraceNode with children from recursion.
  10. Return (nodes, any_truncated).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Mechanic -> KnowledgeGraph | Mechanics access graph only through MechanicContext; no direct NetworkX access |
| Mutation data -> Matcher evaluation | Matchers evaluate mutation data that could contain arbitrary strings |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Elevation of Privilege | MechanicContext | mitigate | MechanicContext stores graph as `self._graph` (private). Mechanics should not access `ctx._graph` directly. No sandboxing in v1 (accepted per project decision) but the API boundary is established. |
| T-02-02 | Tampering | ChainExecutionEngine | mitigate | Max depth limit (default 10) prevents runaway chains. Cycle detection via (mechanic_id, target) pairs prevents infinite loops. Both configurable but with safe defaults. |
| T-02-03 | Denial of Service | ChainExecutionEngine | mitigate | Max depth limit caps recursive execution. A mechanic producing thousands of mutations per apply() is bounded by graph size. No additional throttle needed for v1 hobby project. |
| T-02-04 | Information Disclosure | MechanicContext | accept | MechanicContext exposes full graph query API. In v1 all mechanics are trusted (hand-written or LLM-generated and reviewed). No per-mechanic access control needed. |
</threat_model>
