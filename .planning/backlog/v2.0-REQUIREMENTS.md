# Requirements — Token World v2.0 (Far-Fetched)

*Stub for the milestone after v1.2. These items were called "very very
far fetched" by the user in session 6 — they mostly require multi-agent
engine support, distributed/transactional primitives, or commerce
that v1 doesn't exercise. Sourced from MORNING-HANDOFF §I's
"firmly v2" bucket + CLAUDE.md "Out of Scope" reaffirmations +
`.planning/backlog/phase-03-gap-deferrals.md` (12 items not promoted
to v1.2 Graph Conventions).*

**Milestone vision (draft):** Scale emergence beyond a single resident.
Multiple resident agents with contrasting personalities interacting in
a shared universe, with transactional graph primitives that let their
actions interleave safely, reputation graphs that let trust and
rivalry accumulate, a bird's-eye monitoring surface for the operator,
and commerce patterns that make the "fungible amount" convention earn
its keep. Sandboxed mechanic execution lands here too, once the
hobby-scope security posture gets re-evaluated against the
mechanic-authoring surface area that v1.2 will have grown.

## Active Backlog (candidates for v2.0)

Each item is a terse stub — acceptance criteria will be expanded when
v2.0 is formally opened. Source citations link back to the original
gap / decision.

### REQ-V20-MULTI-01 — Multi-agent simulation engine cutover

Engine supports N>1 resident agents per tick. Today's single-actor
invariant (D-17) is relaxed. Tick order: round-robin or priority-queue
scheduling; observer works per-agent with cross-visibility rules;
conservation invariants extend over multi-agent atomic commits.
Dashboard scaffold already lands in v1.2 (REQ-V12-DASHBOARD-05) — this
is the engine cutover half.

**Source:** v1.0 MULTI-01..03 (deferred); v1.1 D-03 (single-agent
baseline); v1.2 REQ-V12-DASHBOARD-05 (scaffold half)
**Status:** Deferred

### REQ-V20-MULTI-02 — Review agents / bird's-eye monitoring

Once multi-agent lands, an operator-level "review agent" can surface
global consistency metrics across all residents (conservation drift,
character-stability averages, emerging vocabulary overlap, reputation
graph health). Requires multi-agent; cannot exist without it.

**Source:** v1.0 MON-01..03 (deferred); CLAUDE.md Out of Scope section
**Status:** Deferred

### REQ-V20-HARD-01 — Sandboxed mechanic execution (RestrictedPython)

Hobby-scope v1 ships without sandboxing; mechanics execute in the
engine's own Python process. v2 should re-evaluate: if the mechanic
corpus grew materially in v1.2 (overnight authoring runs), the attack
surface of untrusted Python code may warrant the RestrictedPython
integration that was deferred for v1. CVE-2025-22153 remains on the
watchlist.

**Source:** v1.0 HARD-01 (deferred); CLAUDE.md Out of Scope; STATE.md
blockers reference
**Status:** Deferred

## Non-requirements for v2.0

These items are **permanently out of scope** for Token World — not
milestone-delayed, but project-level rejections. They remain here as a
reminder of what the project is and is not.

- **Game adaptation / playable game** — Token World is a simulation, not
  a game. No win conditions, no score loop.
- **Multimodal output (images, maps, 8-bit graphics)** — text-first.
  Graph visualisation is operational (Mermaid + NiceGUI), not player-facing.
- **Civic simulation scenarios (UBI, economics)** — would require a
  mature mechanics ecosystem and domain-specific fidelity that isn't
  the project's purpose.
- **Real-time simulation** — turn-based is simpler and sufficient.
- **Authentication / multi-user** — local hobby project; no remote
  multi-tenancy.
- **Plugin system for mechanics** — the framework *is* the plugin API.
  Every mechanic is already a Python module authored via normal SDLC.
- **Cloud-hosted / public dashboard URL** — dashboard stays local-only.
- **Agent SDK overnight runs** — forbidden per §L cost rails. All
  overnight work uses subagents under the user's Claude subscription
  via `ExternalOperator` file protocol. This is not a future change —
  it is the blessed shape.
